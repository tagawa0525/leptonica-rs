# 回帰テスト網羅性監査・修正計画

## Context

ベンチマーク比較（C vs Rust）で、145テスト中131テストでRustが高速という異常な結果が出た。調査の結果、Rustテストの大半がC版と等価な処理を行っていないことが判明。

具体的な問題:

- **40テストが実行時間0秒**: データ構造のユニットテストが回帰テストを装っている
- **golden比較の欠如**: C版は`regTestWritePixAndCheck`でピクセル単位検証するが、Rust版は`width`/`height`チェックのみ
- **実画像未使用**: 合成データ（10x10等）でのみテストし、C版が使う実画像を処理していない
- **計算量の不一致**: C版が数百回のブレンド/畳み込みを行う箇所で、Rust版は1-2回の呼び出し

## 方針

2段階で監査する:

1. **機械的スクリプト**で関数呼び出し数を集計し、C版との乖離が大きい順にソート（優先順位付け）
2. **LLMによる意味的レビュー**を乖離の大きいものから順に実施（機械的カウントでは見逃す問題を検出）

さらに、Rustの出力がCとbit単位で一致するかを検証し、一致・不一致の両方を記録する。

## Phase 1: 機械的監査スクリプト

`scripts/audit-regression-tests.py` を作成。各テストについて以下を集計:

- C版: `regTestWritePixAndCheck` / `regTestCompareValues` / `regTestComparePix` / `regTestCheckFile` の呼び出し数
- Rust版: `#[test]`関数数 / `#[ignore]`数 / `compare_pix()` 数 / `compare_values()` 数 / `write_pix_and_check()` 数 / `load_test_image()` の有無

乖離スコアを算出（例: `C checks - Rust checks` の比率）し、大きい順にソート。

出力: `docs/porting/regression-test-audit.csv`

## Phase 2: LLMによる全テスト意味的レビュー

> **テスト数について**: `alltests_reg.c` には149テストが列挙されている。
> C版ベンチマークは環境依存のオプションテスト（jp2kio, webpanimio, webpio）を除いた
> 145テストを対象とするが、この監査スクリプトは149テスト全てを対象とする。

Phase 1の乖離スコアが大きいものから順に、サブエージェントで意味的レビューを行う。

### 各テストの判定項目

1. **カバレッジ**: C版の各チェックポイントに対応するRust側のチェックが存在するか
2. **計算等価性**: C版と同じ画像・同じパラメータ・同じ回数の処理を行っているか
3. **検証の質**: 結果をピクセル単位で検証しているか、それとも寸法チェックのみか
4. **未実装依存**: テストに必要だがRustに未実装の関数はあるか（関数名を列挙）

### 実行方法

- 10〜15テストずつバッチにしてサブエージェント（haiku）を並列起動
- 各エージェントはC版とRust版の両方を読み、上記判定を行う

### 出力

`docs/porting/regression-test-audit.md` に以下を記録:

| test | 分類 | C checks | Rust checks | Rust ignored | bit一致 | 欠落チェック | 未実装関数 | 備考 |
| ---- | ---- | -------- | ----------- | ------------ | ------- | ------------ | ---------- | ---- |

**分類**:

- **A: 等価** — C版と同等のチェック数・計算量・検証品質
- **B: 部分的** — 一部のチェックが欠落しているが、必要な関数は存在する → 修正可能
- **C: 不足（未実装依存）** — 必要な関数がRustに存在しない → `#[ignore]`テスト追加 + 計画書記録
- **D: 設計差異** — Rust版の設計上C版と1:1対応しない正当な理由がある → 理由を記載

**bit一致**列の値:

- **一致** — C版goldenとRust版goldenがバイナリ一致
- **差異(fp)** — 浮動小数点の演算順序差による微小な差（許容範囲を明記）
- **差異(alg)** — 意図的に異なるアルゴリズムを採用（理由を明記）
- **差異(bug)** — Rust実装の不具合 → Issue化
- **未検証** — golden生成に必要な機能が未実装、または分類C/Dのため検証不可

### bit一致検証の方法

1. C版テストを `generate` モードで実行し、golden画像を生成
2. Rust版テストを `generate` モードで実行し、golden画像を生成
3. 同一テスト・同一インデックスのファイルをバイナリ比較
4. 結果（一致/不一致とも）を上記の表に記録

## Phase 3: テスト修正

Phase 2で分類Bと判定されたテストを修正する。bit一致検証で差異(bug)が見つかった実装も合わせて修正する。

修正の原則:

1. C版の `*_reg.c` を読み、全チェックポイントを把握する
2. 対応するRust関数が存在するチェックについて、テストを追加する
3. `compare_pix()` / `write_pix_and_check()` を使い、ピクセル単位の正確性を検証する
4. `load_test_image()` で実画像を使用する
5. C版と同等の計算量（ループ回数、画像サイズ）を再現する
6. 既存のユニットテスト的な内容は残しつつ、回帰テスト相当のチェックを追加する

分類Cのテストには `#[ignore = "未実装関数名"]` 付きのテストスケルトンを追加する。

PRの粒度: テストディレクトリ単位（core, io, transform, filter, color, region, recog, morph）

## Phase 4: 検証

1. `cargo test --all-features` で全テスト通過を確認
2. `./scripts/benchmark-regression-145.py rust --runs 1` で `mapped_missing=0` を確認
3. Phase 1-2の監査を再実行し、分類Aの割合が改善されていることを確認
4. bit一致検証を再実行し、監査表を更新
5. ベンチマーク比率が妥当な範囲に収まるテストが増えていることを確認
