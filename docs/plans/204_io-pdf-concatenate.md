# io/pdf: PDF 連結関数の移植（保留）

Status: HOLD
親計画: [031_gap-fill-overall.md](031_gap-fill-overall.md) (項目 G)

## Context

C 版 `pdfio1.c` / `pdfio2.c` の以下 4 関数が Rust に未移植。

| C 関数                     | 行            | 役割                                                        |
| -------------------------- | ------------- | ----------------------------------------------------------- |
| `concatenatePdf`           | pdfio1.c:1996 | ディレクトリ内の単ページ PDF を語彙順で連結してファイル出力 |
| `concatenatePdfToData`     | pdfio1.c:2106 | 上記のメモリ版                                              |
| `ptraConcatenatePdf`       | pdfio1.c:2063 | PDF バイト列の Ptra から連結してファイル出力                |
| `ptraConcatenatePdfToData` | pdfio2.c:329  | 上記のメモリ版 (中核実装)                                   |

## 調査結果

### C 版実装の前提条件

`ptraConcatenatePdfToData` のヘッダコメントに明記されている通り、これらの関数は
**Leptonica が `pdfio2.c` で生成する特定構造の単ページ PDF ファイル限定**で動作する:

```text
(1) This only works with leptonica-formatted single-page pdf files.
    pdf files generated by other programs will have unpredictable
    (and usually bad) results.  The requirements for each pdf file:
      (a) The Catalog and Info objects are the first two.
      (b) Object 3 is Pages
      (c) Object 4 is Page
      (d) The remaining objects are Contents, XObjects, and ColorSpace
```

実装は:

1. 各入力 PDF をパースして trailer/オブジェクト位置 (`parseTrailerPdf`) を抽出
2. 1 ページ目以外は最初の 3 オブジェクト (Catalog/Info/Pages) をスキップ
3. オブジェクト番号を付け直して `substituteObjectNumbers` で参照を更新
4. 全ページの Page オブジェクトを集めた新しい Pages オブジェクトを生成
5. trailer を末尾に再構築

要するに **PDF の低レベルなパース + 構造再構築** が必要。

### pdf-writer 0.14 の機能

`pdf-writer` 0.14 のソース ([docs.rs](https://docs.rs/pdf-writer/0.14.0/pdf_writer/)、
[GitHub](https://github.com/typst/pdf-writer/tree/v0.14.0)) を確認した結果:

- `Pdf::new()` / `Catalog`, `Pages`, `Page` 等のオブジェクト書き込み API
- `Chunk::extend(&Chunk)` で Chunk のマージは可能
- **PDF パーサーは未提供**（writer 専用クレート）

つまり、既存 PDF バイト列をパースして再構築する機能はないため、本タスクの実装には
別途 PDF パーサーが必要。

### Rust の既存 API

```rust
// 複数 Pix → 1 つの multi-page PDF
pub fn write_pdf_multi<W: Write>(
    images: &[&Pix],
    writer: W,
    options: &PdfOptions,
) -> IoResult<()>;

// 複数の画像ファイル → 1 つの multi-page PDF
pub fn write_pdf_from_files<W: Write>(
    paths: &[impl AsRef<std::path::Path>],
    writer: W,
    options: &PdfOptions,
) -> IoResult<()>;
```

C 版 `concatenatePdf` の主な実用シナリオ「Leptonica で 1 ページずつ生成した PDF を
あとでまとめる」は、Rust 版では「最初から `write_pdf_multi` で multi-page PDF を
生成する」ワークフローで代替できる。

### 判断: 保留

- **採用案 C**: 本 PR では `concatenatePdf` 系を実装しない
- 利用者が複数 Pix/画像ファイルから multi-page PDF を作りたい場合は既存の

  `write_pdf_multi` / `write_pdf_from_files` を案内する

- 「既存 PDF ファイル群の連結」が必要になったら、`lopdf` or `pdf` crate を

  オプション feature として追加する案 (A) を別 issue で起票する

`tests/io/pdfio*_reg.rs` の `convertSegmentedFilesToPdf` / "PDF concatenation" 系の
`#[ignore]` テストは、本タスクでは触らずに残す（マッピングが本質的に異なるため、
個別判断が必要）。
